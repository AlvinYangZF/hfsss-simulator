First attempt failed with `QEMU guest did not become reachable over SSH`. Root cause: `/tmp/hfsss_qemu_key` is regenerated by the runner whenever `/tmp` is wiped (macOS reboot does this), but the bundled `guest/cidata.iso` was last rebuilt 2026-04-15 and authorizes the previous runner key only. Resolution applied in this run (state preserved):

- Backed up `guest/cidata.iso` → `guest/cidata.iso.bak` and `guest/cidata/user-data` → `user-data.bak`
- Added `/tmp/hfsss_qemu_key.pub` to `guest/cidata/user-data` `ssh_authorized_keys`
- Added `runcmd:` block that copies `/home/alpine/.ssh/authorized_keys` to `/root/.ssh/authorized_keys` (cloud-init's top-level keys land in the default user, but the runner connects as root@127.0.0.1)

Next time `/tmp` is wiped this will recur. A persistent solution would be to host the SSH key under `~/.ssh/` (or a project-local persistent path) and update `HFSSS_SSH_KEY` default in `scripts/qemu_blackbox/lib/env.sh` accordingly, so cidata never has to be rebuilt.

### 5.1 Failure signature

Every fio failure presents the same way:

- fio prints `io_u error on file /dev/nvme0n1: No data available: write offset=…, buflen=…`
- guest dmesg shows `nvme0n1: I/O Cmd(0x1) @ LBA …, 8 blocks, I/O Error (sct 0x2 / sc 0x80)` followed by `critical medium error … op 0x1:(WRITE)`
- NBD server log shows clean handshake, switches to `Mode: ASYNC PIPELINE (SQ + CQ + 8 FTL workers)` on client connect, no error logged on its side, exits cleanly

The SCT 0x2 / SC 0x80 status from the simulator end maps to NVMe Write Fault — the simulator reported failure to the host, the host kernel surfaced it to fio, fio aborted.

### 5.2 Differential: mt vs async pipeline

The same fio cases (010 – 013) PASS under bare `make qemu-blackbox` (mt + 512 MiB) and FAIL under `make pre-checkin` (async + 2 048 MiB). Two independent variables changed; the dominant suspect is the dispatcher topology, not the capacity, because:

- mt mode runs through a single dispatcher that funnels into the 8 FTL workers
- async mode (`Mode: ASYNC PIPELINE (SQ + CQ + 8 FTL workers)`) runs multiple concurrent dispatchers feeding the same FTL workers
- the failure is a *write fault*, which on the simulator side is produced by the FTL/HAL when concurrent state would otherwise corrupt — i.e. the safety net firing, not a media defect

### 5.3 Relation to prior investigation

This matches the explicit follow-up the 2026-04-18 closeout left open. From `docs/superpowers/plans/2026-04-17-fio-014-015-investigation-closeout.md`:

> Under very high concurrency (`ftl_mfc_repro threads=16`) we still observe an occasional data-integrity mismatch, implying a second race beyond CWB — left as future work because: It does not manifest in the single-dispatcher NBD production path above a negligible rate.

Pre-checkin's async pipeline is the multi-dispatcher case that the closeout did not chase down. The 2 048 MiB exported size widens the LBA range and gives concurrent writers more chance to collide on the same `(channel, plane, block)` tuple, so the second race surfaces stably here rather than as a one-in-many-million event under mt mode at 512 MiB.

### 5.4 Not a regression from PR #111

PR #111 (REQ-045 tier-1 completion queue) only touches `include/controller/channel_worker.h`, `src/controller/channel_worker.c`, `tests/test_channel_worker.c`, and `docs/REQUIREMENT_COVERAGE.md`. The NBD / FTL / HAL hot paths are untouched. The pre-checkin failure is reproducible directly off `e930fce` master (the merge commit), and would reproduce off any commit prior to today as well per the prior closeout's wording.
